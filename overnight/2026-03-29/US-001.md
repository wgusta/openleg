# US-001 — Define Verified Municipality Demand Signal

**Date:** 2026-03-29

**Status:** ✅ Implemented & merged via PR

## Summary

Implemented `compute_municipality_demand_signal()` in the data/insights layer
(`insights_engine.py`). The function aggregates real resident interactions from
the platform database into a per-municipality demand metric that operators can
inspect and reuse via `refresh_all_insights()` or by calling the function
directly.

**Key design decisions:**

| Concept | Approach |
|---|---|
| Municipality grouping | Join `municipalities` table (BFS number) to `buildings` via `city_id = subdomain` |
| Verified demand signals | `verified_buildings`, `recent_signups_90d`, `confirmed_leg_members`, `communities_in_formation`, `meter_data_uploads` |
| Heuristic distinction | `signal_type = "verified"` when ≥1 resident action exists; `"heuristic_only"` for municipalities present only in public-data tables (Sonnendach, ELCOM tariffs) |
| Composite score | Weighted, capped per component (0–100): verified buildings ×2 (max 40), recent signups (max 15), LEG members ×3 (max 30), communities in formation ×5 (max 15) |
| Demand level | `"high"` ≥40, `"medium"` ≥15, `"low"` >0, `"none"` =0 |
